Источник правки — формат нотации задач из проекта [mdtask](https://mdtask.dev/) (сторонний open-source инструмент: спека и таск-бэклог в одном markdown-файле, коммит закрывает задачу только вместе со спекой). Взят **только формат и один процессный gate**, не сам инструмент — иначе это повторило бы ошибку v6, которую сам документ ниже критикует: вводить кастомный внешний механизм там, где хватает соглашения поверх уже существующей структуры (Memory Bank + `.clinerules`).

| Было в v7.0 | Стало в v7.1 | Почему |

| Задачи в `01-detailed-flow.md` и `progress.md` — прозаический список, "первая невыполненная микро-задача" вычисляется эвристически | Единая нотация задач: `- [ ] ID Название #тег !приоритет @blocked_by:ID` | Детерминированный грепаемый формат вместо эвристики. Агенту не нужно "вычислять" первую невыполненную задачу из текста — он читает первый unblocked чекбокс. Формат не требует стороннего CLI, это соглашение внутри уже существующих файлов. |

Явный gate закрывает этот зазор без нового механизма — это ужесточение уже существующего правила `[archived → Obsidian:<ссылка>]`, а не новый процесс. |

## `[v7]` Что изменилось и почему — сводка перед документом

Это не косметическая правка. При проверке v6 на best practices выяснилось, что три крупных механизма дублируют **уже встроенные функции Cline**, а не являются вашей добавленной ценностью. Их кастомные версии не просто избыточны — они менее надёжны, чем системные аналоги, потому что реализованы через текстовые договорённости с моделью вместо системных ограничений инструмента.

| Было в v6 | Заменено на | Почему |
|---|---|---|
| Кастомный гейт "слово APPROVED в чате" | Нативный **Plan Mode / Act Mode** Cline | В Plan Mode Cline физически не может менять файлы — это ограничение на уровне инструмента, а не просьба к модели. Текстовый гейт модель может неверно интерпретировать в длинном контексте; системный — нет. |
| Ролевая матрица с ручной оркестрацией qwen/Kimi через раздельные промпты | Настройка `planActSeparateModelsSetting` (раздельные модели для Plan и Act) | Это ровно ваш кейс — reasoning-модель для плана, другая для исполнения — уже встроено как одна галочка в настройках Cline. Ручная оркестрация поверх неё не добавляет ничего, кроме риска ошибиться в передаче контекста между "ролями". |
| Урезанный Memory Bank (4 файла) | Канонический Memory Bank Cline (6 файлов) + `decisions.md` как ваше дополнение | Официальный паттерн `.clinerules/memory-bank.md` уже опубликован командой Cline, обкатан сообществом и точно указывает, какие файлы обязательны и в каком порядке их читать. Ваша версия v6 пропускала `productContext.md` и `systemPatterns.md` — а туда как раз должны попадать архитектурные инварианты, которые вы размазали по Шагу 2. |
| Post-Mortem сразу после 3-й неудачи | Откат к **Checkpoint** перед Post-Mortem | У Cline уже есть теневой git с моментальными снимками после каждого действия агента. Дешевле откатиться и попробовать другую формулировку, чем сразу писать разбор полётов. |

Оставлено без изменений, потому что не дублирует ничего готового: **Karpathy Loop** (стресс-тест конкурентности под вашу схему БД) и **Shadow Evaluator / Антагонист** (adversarial review в отдельном контексте) — таких функций в Cline нет, это ваша реальная добавленная стоимость.

## 1. Архитектурная Матрица Ролей (The Team)

- **Plan-модель (проектирование + PM-функции)**: `qwen3.6-35b-a3b` через NeuralDeep Hub (`https://api.neuraldeep.ru/v1`), настроена как модель **Plan Mode** в Cline. Контроль прогресса, выбор подзадач, построение планов, контроль инвариантов.
- **Act-модель (исполнение)**: `kimi-k2.6`, настроена как модель **Act Mode** в Cline. Генерация кода, TS-типов, исправление ошибок компиляции.
- **Shadow Evaluator / Антагонист** (верификация, вне Plan/Act цикла): `qwen3.6-35b-a3b` в изолированном чистом чате. Установка: «Найти баг, race condition или архитектурную лазейку любой ценой. Быть предвзятым и деструктивным».

### `[v7]` Почему убрана "ролевая матрица" из четырёх пунктов

В v6 PM, Doc Architect и Antagonist были описаны как три отдельные "роли" одной модели с ручным переключением между чатами. По факту PM и Doc Architect — это одна и та же функция (Plan Mode), Cline уже разделяет её от Act Mode системно через `planActSeparateModelsSetting`, где каждому режиму назначается свой независимый API-конфиг (модель, провайдер, параметры). Ручное описание этого как "матрицы ролей" не добавляло контроля — оно добавляло риск, что при копировании промптов между "ролями" контекст потеряется или исказится.

**Настройка в Cline:** Settings → включить `Use different models for Plan and Act` → выбрать qwen3.6-35b-a3b для Plan, kimi-k2.6 для Act. **`[v7.2]` TODO:** название галочки/поля во внутреннем API Cline (`planActSeparateModelsSetting`) сверить с фактическим UI на вашей установленной версии — у вас уже есть источник истины (реальный интерфейс), сверка не требует поиска в документации.

**Известный риск:** если оба провайдера настроены на одном "provider slot" (например, оба через NeuralDeep как один OpenAI-совместимый провайдер), в некоторых версиях Cline встречался баг, что переключение режимов не подхватывает разные модели корректно. Проверьте перед тем, как полагаться на это в реальной работе — попробуйте переключиться Plan → Act на тестовой задаче и убедитесь, что в логе виден вызов именно kimi-k2.6, а не повторный вызов qwen.

### `[v7]` Известное ограничение (перенесено из v6, актуально по-прежнему)

Shadow Evaluator — та же модель (qwen), что и Plan. Разрыв контекста снижает confirmation bias, но не устраняет системные слепые пятна модели: если qwen системно не видит определённый класс уязвимостей, она не увидит их ни как автор плана, ни как антагонист.

**Правило эскалации (без изменений с v6):** для задач, затрагивающих auth, квоты (`INV-07`), блокировку версий (`INV-09`) или любые DDL-миграции — на Шаге 6 Разработчик вручную прогоняет `git diff` через **вторую модель другого провайдера** (Kimi k2.6 или иной доступный вариант) как дополнительное мнение. Не автоматизировано намеренно — только для помеченных критических задач.

### `[v7.3]` Обязательный pre-flight чек перед первым запуском на проекте

Перед Шагом 0 самой первой задачи (обычно — старт Этапа 1) Разработчик вручную выполняет одноразовую проверку:

1. Открыть тестовую задачу в Cline.
2. Переключить Plan → Act.
3. Проверить в логе запроса (или статусбаре Cline, если отображает активную модель), какая модель реально вызвана в Act Mode.
4. Если вызывается `qwen3.6-35b-a3b` вместо `kimi-k2.6` — это баг конфигурации provider slot: нужны два раздельных provider slot в настройках Cline, даже если оба указывают на один `base_url` (`https://api.neuraldeep.ru/v1`).

Результат фиксируется в `activeContext.md` одной строкой («Act-модель подтверждена: kimi-k2.6, дата») и не повторяется на каждой сессии — только один раз до начала Этапа 1, и повторно только если менялась конфигурация провайдеров в Cline.

Не пропускать эту проверку. При едином provider slot для Plan и Act документированный риск — тихая подмена модели в Act Mode без какого-либо сигнала об ошибке. Если это произойдёт незамеченным, Plan, Act и Shadow Evaluator (§1, установка «Антагонист») окажутся одной и той же моделью на всех трёх ролях — весь смысл разделения (независимый взгляд на код на Шаге 6) исчезнет молча, при этом каждая роль формально отработает без видимой ошибки.

### 2.2. Memory Bank (Внутренний Контур Проекта) — рабочий кэш сессии

### `[v7]` Переход на канонический набор файлов Cline

**Было в v6:** `projectBrief.md`, `activeContext.md`, `progress.md`, `decisions.md` (4 файла).

**Стало в v7** — официальная структура `.clinerules/memory-bank.md` от команды Cline, плюс один ваш файл сверху:

```
memory-bank/
├── projectbrief.md      # Фундамент: цели, требования, объём проекта
├── productContext.md    # [v7, новый] Зачем проект существует, какие проблемы решает
├── systemPatterns.md    # [v7, новый] Архитектура, паттерны, critical implementation paths
├── techContext.md       # [v7, новый] Стек, окружение, зависимости, ограничения
├── activeContext.md     # Текущий фокус, последние изменения, следующие шаги, retry_count [v7.2]
├── progress.md          # Что работает, что осталось, известные проблемы
└── decisions.md         # [ваше дополнение] ADR-черновики текущей сессии
```

**Почему добавлены три файла:** в v6 архитектурные инварианты и паттерны конкурентности были свалены в план Шага 2 и нигде не сохранялись между сессиями отдельно от Obsidian. Канонический `systemPatterns.md` — это ровно то место, куда должны попадать "критические implementation paths" (в терминологии официального паттерна) — то есть ваши правила вроде `INSERT ... ON CONFLICT` вместо цепочек проверок. `productContext.md` и `techContext.md` реже меняются, но обязательны к прочтению каждый раз — без них агент каждый раз заново выясняет "зачем это вообще нужно" и "что уже есть в стеке".

**`[v7.3]` Разрешение конфликта Master ↔ Memory Bank:** если `systemPatterns.md` противоречит текущей версии Architecture Master (например, инвариант обновился, а локальный кэш не подтянулся) — Master побеждает безусловно, без ревью человеком. Агент обновляет `systemPatterns.md` под актуальную версию Master, а не пытается согласовать оба источника. Расхождение логируется в `decisions.md` как «не архитектурное» (кэш устарел, это не новое решение) — критерий архитектурного решения из §2.2 здесь не применяется, так как ничего не меняется, только синхронизируется.

**Обязательное правило (из официального паттерна, отсутствовало явно в v6):** агент читает **ВСЕ** файлы Memory Bank в начале **каждой** задачи, не только `progress.md`. Пропуск этого шага — не экономия времени, а гарантированная потеря контекста и дублирование уже сделанной работы.

### `[v7.1]` 2.3. Нотация задач в `progress.md` и `01-detailed-flow.md`

Формат заимствован у [mdtask](https://mdtask.dev/) — только синтаксис, без установки самого инструмента как зависимости:

```
- [ ] INV-07 Инкремент квоты через атомарный RPC #db !high @blocked_by:INV-04
      Заменить SELECT+UPDATE на INSERT...ON CONFLICT DO UPDATE, см. Master §1.
- [x] INV-04 Автосоздание agent worker триггером #db !high
```

Правила:
- **ID** — либо `INV-XX` для задач, привязанных к инварианту из `00-core-invariants.md`, либо произвольный префикс проекта для остальных задач (например `UI-01`, `BOT-03`).
- **Тег** (`#db`, `#ui`, `#bot`, `#infra`) — область, к которой относится задача; используется агентом на Шаге 0 для быстрой фильтрации при выборе следующей микро-задачи.
- **Приоритет** (`!high` / `!med` / `!low`) — не заменяет очерёдность этапов в `01-detailed-flow.md`, а разрешает конфликт внутри одного этапа, если задач с одинаковым статусом несколько.
- **`@blocked_by:ID`** — явная зависимость. Агент на Шаге 0 не берёт задачу с непустым `@blocked_by`, пока указанный ID не отмечен `[x]`.
- Однострочное описание под задачей — не обязательно, если пункт самоочевиден из названия; но обязательно, если задача — INV-привязанная (тогда описание кратко объясняет *что именно* делать, отсылая к соответствующему параграфу Architecture Master, а не дублируя его).

Это заменяет прозаическое "вычисление первой невыполненной микро-задачи" (v7.0, Шаг 0, п.3) на детерминированный разбор: первая строка с `- [ ]` без непустого невыполненного `@blocked_by` — и есть следующая задача.

**`[v7.2]` Разграничение задач по этапам:** нотация не вводит отдельное поле `@stage`/`@phase`. Активный этап определяется по секции (заголовку) `01-detailed-flow.md`, в которой физически расположена задача — Шаг 0 п.3 ищет первую подходящую строку `- [ ]` внутри секции активного этапа, а не по всему файлу. **Требует проверки:** это работает только если задачи в файле действительно разбиты по секциям-заголовкам этапов. Если фактическая структура `01-detailed-flow.md` перемешивает задачи разных этапов без такого разделения — ввести `@stage:N` как дополнение к нотации; до этой проверки поле не добавляется, чтобы не усложнять формат ради проблемы, которая может не существовать.

### Разграничение ролей Memory Bank и Obsidian (без изменений с v6)

`memory-bank/` — временный кэш, не альтернативный источник правды. `decisions.md` копит сырые решения по ходу сессии. На **Шаге 7** значимые решения переносятся в постоянные ADR Obsidian; после переноса запись в `decisions.md` помечается `[archived → Obsidian:<ссылка>]`, не удаляется.

Если решение осталось только в `decisions.md` без переноса — оно неподтверждено и не может быть основанием для будущих решений.

**`[v7.2]` Формат записи `decisions.md`:** каждая запись — заголовок с ID задачи и датой, тип записи (`архитектурное` / `не архитектурное` / `вердикт-антагонист`), краткое содержание, и для архитектурных — статус переноса (`[pending]` / `[archived → Obsidian:<ссылка>]` / `[отложено: <причина>]`). Запись типа `вердикт-антагонист` обязательна после каждого прогона Шага 6 независимо от исхода (принят/отклонён) — см. правило в Шаге 6.

## 3. Автономный Операционный Цикл Задачи (The Flow)

```
 [Шаг 0: PM Фаза]   ──> Читает ВСЕ файлы memory-bank/, проверяет окружение, обновляет activeContext.md
        │
 [Шаг 1: Контекст]  ──> Читает нужную Obsidian MOC-карту и спецификации кода
        │
 [PLAN MODE]        ──> Qwen строит технический план + план симуляции
        │               (файлы менять не может — системное ограничение Cline)
        │               Человек переключает тумблер Plan → Act, когда план принят
        │
 [ACT MODE]         ──> Kimi пишет код + генерирует TS-типы Supabase
        │               Checkpoint создаётся автоматически после каждого изменения
        │
 [Шаг 4: CLI Loop]  ──> Локальная компиляция tsc / deno check (Retry limit = 3)
        │               При неудаче на 3-й попытке → откат к Checkpoint, см. Раздел 4
        │
 [Шаг 5: Karpathy]  ──> Локальный стресс-симулятор (конкурентные RPC / гонки квот)
        │
 [Шаг 6: Черных]    ──> Агент-Антагонист в чистом чате атакует git diff
        │               [если критическая задача → доп. проверка второй моделью]
        │
 [Шаг 7: Финал]     ──> Приёмка человеком ──> перенос decisions.md в Obsidian ──>
        │               progress.md обновлён ──> цикл замыкается
```

### `[v7]` Почему исчез отдельный "Шаг 2: Plan" и "Шаг 3: Act" как пронумерованные шаги

В v5/v6 это были шаги вашего кастомного протокола с ручным гейтом APPROVED. В v7 это те же по смыслу фазы, но реализованные через системные режимы Cline — Plan Mode и Act Mode — а не через текстовую договорённость. Нумерация с остальными шагами (0, 1, 4, 5, 6, 7) сохранена для совместимости с вашими существующими MOC-ссылками и `01-detailed-flow.md`, но сам переход Plan→Act больше не требует ритуальной фразы, потому что физически невозможен, пока вы не переключите режим.

### Шаг 0: Автономная PM-Фаза (Orchestration)

При старте сессии Cline самостоятельно:

1. **`[v7]`** Читает **все** файлы `memory-bank/` (не только `progress.md`) — согласно каноническому паттерну.
2. Анализирует раздел декомпозиции в `01-detailed-flow.md`.
3. **`[v7.1]`** Находит первую строку `- [ ]` активного этапа, у которой `@blocked_by` пуст или все указанные ID уже отмечены `[x]` — согласно нотации §2.3. Это заменяет прежнее эвристическое "вычисление" явным разбором формата.
4. Проверяет фактическую доступность окружения (без изменений с v6):
   - MCP-серверы (`pg-aiguide`, Figma) — health-check; если недоступны, явно сообщает и продолжает без них.
   - Docker / Supabase CLI — статус локального инстанса.
5. Перезаписывает `activeContext.md`: название задачи, этап, ссылка на MOC.
6. Выводит в чат: «Взял в работу задачу №X [Название]. Окружение проверено: [статус]. Перехожу к сбору контекста».

### Шаг 1: Автономный сбор контекста

Агент открывает MOC-карту из `activeContext.md`, вычитывает документы этапа, проверяет файловую структуру проекта.

**`[v7.2]` Правило выбора MOC-карты:** имя карты выводится из тега задачи (`#db` / `#ui` / `#bot` / `#infra`), назначенного согласно нотации §2.3, по соответствию `#тег → MOC-<область>-*.md` (например `#db` → `MOC-DB-Migrations.md`). Тег фиксируется в `activeContext.md` уже на Шаге 0 — на Шаге 1 он используется, а не выбирается заново. Это закрывает разрыв между Шагом 0 (тег назначен) и прежней формулировкой «открывает MOC-карту» без указания источника имени, не вводя новый механизм — используется уже существующая нотация задач.

### Plan Mode: Проектирование

`qwen3.6-35b-a3b` в Plan Mode строит технический сценарий. План фиксирует:

- Изменения DDL / структуры таблиц.
- Влияние на TypeScript-интерфейсы.
- Риски нарушения инвариантов `00-core-invariants.md`.
- Сценарий стресс-тестирования логики.
- Явная пометка: относится ли задача к критическому классу (auth/квоты/версии/DDL), требующему второй модели на Шаге 6.

**`[v7]` Гейт реализован системно:** Cline в Plan Mode не имеет доступа к инструментам изменения файлов и выполнения команд — это ограничение уровня приложения, а не просьба к модели. Переход в Act Mode — явное действие человека (переключатель в интерфейсе Cline), не текстовая фраза.

**Для многофайловых или архитектурно значимых задач `[v7.2]`:** Plan-модель создаёт отдельный файл плана `.clinerules/plans/<task-id>.md` с детальным техническим сценарием, на который можно ссылаться в течение всей многосессионной работы. Это соглашение поверх уже существующей структуры `.clinerules/`, не встроенная команда Cline и не зависимость от стороннего инструмента — закрывает то, что в v5/v6 пытался решить громоздкий текстовый формат плана Шага 2, без повторения ошибки v6 (кастомный механизм там, где хватает соглашения).

### Act Mode: Реализация

`kimi-k2.6` в Act Mode последовательно вносит изменения. Если задача затрагивает миграции БД — сразу после изменения DDL обязательна команда:

```bash
supabase gen types typescript --local > lib/types/database.types.ts
```

**`[v7]` Checkpoints включены по умолчанию:** после каждого изменения файла или выполнения команды Cline автоматически создаёт снимок в теневом git-репозитории, не затрагивающем ваш основной git. Это не требует настройки — важно только не выключать эту опцию в Settings, если репозиторий не аномально большой.

### Шаг 4: Локальный CLI-фильтр (Closed CLI Loop)

- Next.js/Node: `npm run check:node` (`tsc --noEmit`)
- Edge Functions: `npm run check:deno` (`deno check`)
- **`[v7.2]` Обязательный подшаг перед переходом к Шагу 5:** `npm test` (unit/integration) — покрывает FK-констрейнты, RLS-политики и структурные инварианты, не относящиеся к конкурентности (см. таблицу привязки инвариантов ниже). Не альтернатива компиляции, а обязательный третий подшаг наравне с ней. CI на коммите (после Шага 7) остаётся как повторная проверка постфактум, но не заменяет локальный прогон — на момент коммита retry-логика Шага 4 уже неприменима.
- Retry Limit = 3 (общий счётчик `retry_count` — см. правило ниже — считает неудачи по всем трём подшагам вместе: компиляция и тесты входят в один цикл Closed CLI Loop).

**`[v7]` Новое правило перед 3-й попыткой:** прежде чем тратить третью попытку правки поверх уже накопленных изменений, откатиться к Checkpoint, сохранённому перед первой неудачной попыткой (`Restore Files & Task`), и попробовать иную формулировку задачи. Если и это не помогает — переход к Post-Mortem (Раздел 4).

**`[v7.2]` Поле `retry_count` в `activeContext.md`:** увеличивается на 1 при каждой неудачной попытке `tsc`/`deno check`/`npm test` в рамках Шага 4. Сбрасывается в 0 при: (а) переходе к новой задаче на Шаге 0, или (б) откате к Checkpoint перед новой формулировкой (см. правило выше) — не только «при успехе», поскольку «успех» на этом шаге неоднозначен до прохождения и компиляции, и тестов (см. следующее правило). Достижение `retry_count = 3` без промежуточного сброса — триггер отката к Checkpoint; повторное достижение 3 после отката без дальнейшего успеха — триггер Post-Mortem (Раздел 4).

**Почему:** правка поверх уже сломанного состояния имеет тенденцию накапливать побочные ошибки — то, что в пояснительной записке названо "паническим кодингом". Откат к чистому чекпоинту перед новой попыткой дешевле, чем разбираться, что из трёх слоёв правок стало причиной проблемы.

### Шаг 5: Стресс-симуляция среды (Karpathy Loop Extension)

Без изменений с v5/v6 — не дублирует ничего готового в Cline.

```bash
npm run test:stress
```

Логика считается рабочей, если симулятор подтверждает отсутствие race conditions при 50 параллельных запросах к одной сущности.

### Таблица привязки инвариантов к шагам верификации

**`[v7.3]` Активное подмножество по этапам:** таблица ниже покрывает инварианты всего проекта, не только текущего этапа. На Шаге 0 при старте задачи агент сверяет, какие из перечисленных инвариантов физически проверяемы на активном этапе — по факту существования задействованных таблиц (например, INV-12 требует `assignment_history`, §6.14 Master — недоступна до Этапа 8; INV-13 требует `task_relations`, §6.16 — недоступна до Этапа 8–9 согласно Relational Context Layer). Активное подмножество фиксируется в `activeContext.md` при обновлении на Шаге 0. Попытка прогнать Karpathy Loop или Antagonist-проверку против несуществующей таблицы — не ошибка агента, а сигнал, что взятая задача не соответствует заявленному этапу.

`[v7]` Дозаполнено конкурентно-релевантными инвариантами из `onitask_Architecture_Master_.md` §1 (актуальный список — INV-01…INV-16). ### Шаг 6: Проверка Антагонистом (Shadow Verification / Черных Loop)

Без изменений с v5/v6 — не дублирует ничего готового в Cline. Независимый контекст верификатора получает чистый `git diff --staged` и установку:

«Ты — Senior QA Automation и этичный хакер. Твоя единственная цель — доказать, что автор кода допустил ошибку, пропустил уязвимость или нарушил инварианты `00-core-invariants.md`. Найди способ сломать эту логику. Будь предвзят и деструктивен».

Если задача помечена как критическая — дополнительно прогоняется через вторую модель/провайдера.

**`[v7]` Дополнение:** перед ручным анализом diff Антагонистом можно (не обязательно) прогнать `/explain-changes` на том же diff — встроенная команда Cline, которая даёт объяснение изменений на основе git-ссылок независимо от истории разговора. Это не заменяет адверсариальную роль, но может ускорить понимание объёма изменений перед атакой на них.

Если Антагонист аргументированно находит архитектурную дыру — код отклоняется, сессия возвращается на этап Plan Mode.

**`[v7.2]` Ручной шаг и обязательная фиксация вердикта:** открытие изолированного чистого чата для Антагониста — ручное действие Разработчика (новый чат в Cline с независимым контекстом), не автоматизация инструмента. По завершении разбора вердикт (принят/отклонён, ключевая аргументация, ссылка на затронутый инвариант при наличии) обязательно фиксируется отдельной записью в `decisions.md` по формату §2.2 — независимо от того, породила ли задача архитектурное решение по критерию ниже.

### `[v7.2]` Критерий «архитектурное решение»

Задача считается породившей архитектурное решение (и, соответственно, требующей переноса ADR на Шаге 7), если выполняется хотя бы одно из:

- (а) меняется DDL или структура схемы БД;
- (б) меняется публичный контракт MCP tool (сигнатура, `allowed_tools` scope, формат ответа);
- (в) вводится новый инвариант или меняется формулировка существующего из `00-core-invariants.md`;
- (г) меняется алгоритм разрешения конкурентности (retry-стратегия, блокировка, optimistic-lock подход);
- (д) решение противоречит или явно уточняет ранее зафиксированный ADR.

Если ни один пункт не выполняется — перенос ADR не требуется, достаточно записи в `decisions.md` с меткой «не архитектурное» без дальнейшего переноса.

### Шаг 7: Финализация и Переход

**`[v7.1]` Gate-пункт 0 (проверяется до пунктов 1–4, атомарность коммита):** задача не считается закрытой, пока в одном коммите не обновлены одновременно (а) код, (б) статус `[X]` в `progress.md`, и (в) — если задача породила архитектурное решение по критерию выше — соответствующий ADR в Obsidian, а не отложенная запись в `decisions.md`. Если условие (в) не выполняется в момент коммита, агент либо переносит решение сейчас, либо явно фиксирует в `decisions.md` причину отсрочки — молчаливый пропуск запрещён.

Далее без изменений с v6:

1. Отмечает задачу как выполненную `[X]` в `progress.md`.
2. Переносит значимые решения из `decisions.md` в постоянные ADR Obsidian; помечает `[archived → Obsidian:<ссылка>]`.
3. Очищает поле текущей задачи в `activeContext.md`.
4. Если лимит токенов позволяет — предлагает следующую подзадачу, запуская Шаг 0 заново.

## 4. Протокол Обработки Критических Ошибок (Post-Mortem / Karpathy Loop)

Если Closed CLI Loop (Шаг 4) или Антагонист (Шаг 6) заворачивают изменения 3 раза подряд **и откат к Checkpoint с иной формулировкой (см. Шаг 4) не помог** — агент прекращает попытки исправления «в лоб» и генерирует `failed_loop_reason.md`:

- **Первопричина (Root Cause):** какое системное/рантайм-ограничение не учёл исходный план?
- **Архитектурный просчёт:** какой инвариант был проигнорирован?
- **Инъекция в правила (Fix Action):** какое точечное правило (❌ Плохо / ✅ Хорошо) внести в `.clinerules`, чтобы предотвратить зацикливание?

После отчёта Разработчик очищает контекст (Reset Context в Cline), вносит правило, сессия инициализируется заново.

**`[v7]` Уточнённая последовательность:** 3 неудачи → откат к Checkpoint + новая формулировка → если снова неудача → Post-Mortem. Это на один шаг дороже, чем в v6, но дешевле, чем полный цикл Post-Mortem на проблеме, которая решалась бы простым откатом.

Уточнение из v6 остаётся в силе: если причиной неудач была недоступность окружения (MCP/Docker), это не архитектурная ошибка — Post-Mortem не запускается, агент возвращается к проверке окружения из Шага 0.

## ПОЯСНИТЕЛЬНАЯ ЗАПИСКА: ПОЧЕМУ ЭТИ ИЗМЕНЕНИЯ НЕОБХОДИМЫ

### 1. Зачем нужен «Протокол Автономности» (Шаги 0 и 7)?

Без него каждый заход в Cline тратит контекстное окно на объяснения "где мы остановились". Memory Bank как жёсткий диск: агент считывает весь Memory Bank, понимает где находится, обновляет `activeContext.md`, идёт работать.

### 2. Зачем нужна «Стресс-симуляция» (Шаг 5, Karpathy Loop)?

Стандартные тесты проверяют «зелёный путь». Onitask работает в Telegram-среде с пачками запросов и платными нейро-квотами. Код может пройти `tsc` и намертво заблокировать БД при первой конкурентной нагрузке. Принудительный микро-симулятор гонки данных заставляет проектировать с расчётом на конкурентность.

### 3. Зачем нужен «Агент-Антагонист» (Шаг 6, Черных Loop)?

Если модель сама пишет и сама проверяет код в одной сессии — confirmation bias подтверждает чек-лист формально. Разрыв контекста и смена роли на Хакера-Аудитора заставляют ту же LLM искать уязвимости с позиции критического проверяющего. Это снижает bias за счёт смены контекста, но не устраняет системные слепые пятна модели — отсюда правило эскалации на вторую модель для критических задач.

### 4. Зачем нужна «Ретроспектива ошибок» (Раздел 4, Post-Mortem)?

Зацикленная модель начинает генерировать случайные изменения в надежде угадать решение — это выжигает токены и ломает кодовую базу. Остановка и анализ переводят модель из «панического кодинга» в аналитику.

**`[v7]`** Добавлен промежуточный откат к Checkpoint перед полноценным Post-Mortem — не всякая 3-я неудача является архитектурной проблемой; часть из них решается просто более чистым стартом без накопленных побочных правок.

### `[v7]` 5. Зачем вообще менять v6 на нативные механизмы Cline, если v6 работал бы?

Три причины:

1. **Надёжность.** Системное ограничение ("модель физически не может писать файлы в Plan Mode") не может быть проигнорировано моделью в длинном контексте. Текстовая договорённость ("напиши только после APPROVED") теоретически может — особенно если контекст раздувается и более раннее условие "вымывается" из внимания модели.
2. **Меньше движущихся частей = меньше мест для ошибки.** Каждый кастомный протокол поверх готового инструмента — это ещё одна вещь, которая может разойтись с реальным поведением модели. Если Cline уже умеет разделять модели по режимам, ваша ручная имитация того же самого — лишний код без выигрыша.
3. **Совместимость с экосистемой.** Официальный Memory Bank pattern уже документирован, известен сообществу, и если вы когда-то захотите поделиться `.clinerules/` или найти помощь по нему — вы найдёте её именно для канонической структуры, а не для вашей модифицированной версии с другими именами файлов.

### `[v7.1]` 6. Зачем нотация задач из mdtask, если он не тащится целиком?

Один принцип, два вывода:

1. **Формат — не инструмент.** `mdtask` как CLI решает задачу для repo-only проектов без БД: у него нет ни Supabase, ни real-time, ни Worker abstraction, которые есть у onitask-продукта. Но синтаксис задачи (`ID #тег !приоритет @blocked_by`) — это просто текстовое соглашение, которое можно принять внутри уже существующих файлов, не добавляя зависимость. Установка стороннего CLI ради формата чекбоксов повторила бы ту же ошибку, что критикуется в сводке v7 наверху: кастомный механизм там, где хватает соглашения.
2. **Gate на атомарность коммита — не новый процесс, а явная формулировка того, что подразумевалось.** У v7.0 уже было правило `[archived → Obsidian:<ссылка>]`, но не было проверки *в момент коммита*, что перенос действительно случился. mdtask как философия ("задача не закрыта, пока код и спека не обновлены вместе") просто заставляет сформулировать этот gate явно, а не полагаться на то, что агент "не забудет" на Шаге 7.
